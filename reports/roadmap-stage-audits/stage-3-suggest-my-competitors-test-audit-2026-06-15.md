# Stage 3 — Suggest My Competitors: Test Audit Report

**Date:** 2026-06-15
**Stage:** Stage 3 — Suggest My Competitors
**Goal:** Remove the activation blocker for users who don't know their competitors, by suggesting competitor URLs from the business's name/industry/website — with explicit user confirmation before anything is saved, and manual URL entry fully preserved.

---

## Summary of Changes

Stage 3 adds an assisted competitor-discovery path alongside (never replacing) manual entry. Suggestions are generated by Claude, validated server-side with the **same** URL validator as manual entry, and surfaced for the user to confirm. Nothing is persisted until the user adds suggestions to the form and saves through the existing validated save path.

### Design decisions

1. **No new DB schema / no schema change.** Suggestions are transient. They live in client state until the user confirms, then flow into the existing `businesses`/`competitors` save path. This satisfies the roadmap's "do not store unconfirmed suggestions as competitors."

2. **Report-generation AI contract untouched.** Suggestion logic lives in dedicated modules (`lib/competitorSuggestions.ts`, `lib/suggestCompetitors.ts`), not `lib/ai.ts`. The report contract is unchanged.

3. **Pure parser split out for testability.** Following the Stage 2 `reportSources` pattern, the JSON parser (`parseCompetitorSuggestions`) lives in an import-light module so it can be unit tested in the VM without the Anthropic client.

4. **Suggestions reuse manual validation.** The suggest route runs every suggested URL through `validateCompetitorUrl` (the exact validator manual entries use), so SSRF/private-IP/credential protections apply identically. Invalid suggestions, the user's own website, and duplicates are dropped. Capped at 5.

5. **Confirmation is mandatory.** Suggestions render as a checklist; only "Add selected" moves them into the competitor inputs, and even then they're not saved until the user clicks "Save business details." Adding suggestions never triggers a save (verified by test).

### What changed and why

1. **`lib/competitorSuggestions.ts` (new)** — Pure types + `parseCompetitorSuggestions(rawText)`. **Extracts the first bracketed JSON array** found in the model output (via a `[...]` match, tolerating surrounding prose/markdown fences), then `JSON.parse`s that extracted text. It does **not** strictly validate the whole top-level model output — it throws only when no `[...]` array can be extracted, or when the extracted text is malformed/not an array. Individual malformed items are skipped (items without a usable `url` are dropped); caps at `MAX_SUGGESTIONS` (5). Also exports `normalizeHost` and `dedupeSuggestionsByHost` (added post-review). No runtime imports → unit-testable.

2. **`lib/suggestCompetitors.ts` (new)** — Calls Claude (Haiku default, `AI_MODEL` override) with a prompt that asks for 3–5 real competitors as JSON, then delegates parsing to `parseCompetitorSuggestions`. Separate from `lib/ai.ts`.

3. **`app/api/competitors/suggest/route.ts` (new)** — Auth-gated POST. Requires `industry`. **Applies a per-user cost guard (`checkRateLimit`) before the Anthropic call** — 10 requests/hour, friendly 429 on exceed. Calls `suggestCompetitors`, validates each suggested URL with `validateCompetitorUrl`, then `dedupeSuggestionsByHost` collapses same-host URLs, drops the user's own host, and caps at 5; returns `{ suggestions }`. **No DB writes, no scraping, no report generation.** AI failure returns a friendly 502 telling the user to try again or add manually.

4. **`components/BusinessForm.tsx`** — Added a "Suggest competitors" button + helper text, a suggestions panel (checkbox list with name/URL/reason, labelled "AI suggestions, not verified competitors"), and confirm/dismiss controls. "Add selected" merges chosen URLs into the existing competitor inputs (fills empties, dedupes, caps at 5, leaves a spare slot) and shows a notice reminding the user to save. Manual inputs, add/remove competitor, and the save flow are unchanged.

5. **`scripts/stage-3-suggestions.test.mjs` (new)** — 9 tests (behavioral + structural).

### Post-review corrections (Stage 3 findings)

1. **Per-user cost guard before the AI call.** `lib/rateLimit.ts` adds a lightweight sliding-window limiter: a pure `evaluateRateLimit(timestamps, limit, windowMs, now)` plus an in-memory `checkRateLimit(key, …)` wrapper. The suggest route calls `checkRateLimit(\`suggest:${user.id}\`, 10, 1h)` **before** `suggestCompetitors`, returning a friendly **429** on exceed. No new schema — there is no suggestion storage to count, and the report-generation DB-count pattern would conflate report and suggestion activity, so a purpose-built in-memory guard was chosen. It is per-process (not shared across serverless instances) — a soft cost guard, not a hard quota (documented in Remaining Risks).

2. **Dedupe by host, not exact URL.** Host normalization + dedupe moved into the pure, testable `dedupeSuggestionsByHost` (with `normalizeHost`). `https://rival.com` and `https://rival.com/pricing` now collapse to one suggestion (first wins). Own-site filtering and the cap at 5 are preserved. The route's duplicated `safeHost` helper was removed in favour of the shared `normalizeHost`.

3. **Audit wording on the parser corrected** — see the `lib/competitorSuggestions.ts` description above: the parser extracts the first bracketed array and throws only when none can be extracted or the extracted array is malformed; it does not strictly validate the whole top-level model output.

### Stage 3 automated coverage (`scripts/stage-3-suggestions.test.mjs`, 14 tests, all passing)

**Behavioral — `parseCompetitorSuggestions`:**
1. Parses a clean JSON array into typed suggestions.
2. Extracts the array even with surrounding text / markdown fences.
3. Drops entries without a usable `url` (missing or blank).
4. Caps the list at 5.
5. Throws when no array can be extracted, or the extracted array is malformed.

**Behavioral — rate limiter (`evaluateRateLimit`):**
6. Allows hits under the limit; blocks at the limit and reports a positive retry delay.
7. Prunes timestamps outside the window (expired hits don't count).

**Behavioral — host dedupe (`dedupeSuggestionsByHost`):**
8. Collapses same-host URLs (incl. `www.` and path variants) to the first, and drops the own site.
9. Caps the deduped result at 5.

**Structural:**
10. Suggest route requires auth and an industry; reuses `validateCompetitorUrl`; dedupes by host via `dedupeSuggestionsByHost(validated, ownHost, MAX_RETURNED)`; **does not** call `replace_competitors`, `.insert(`, `scrapeUrl`, or `generateCompetitorReport`; returns `suggestions`.
11. Suggest route applies `checkRateLimit(\`suggest:${user.id}\`, …)` returning **429**, and the guard runs **before** `await suggestCompetitors(` (index-ordered assertion).
12. Business form offers "Suggest competitors", calls `/api/competitors/suggest`, and keeps manual inputs + "Add competitor".
13. Business form requires confirmation (`addSelectedSuggestions` / "Add selected" / `toggleSuggestion`), labels suggestions "not verified competitors", and adding suggestions does not trigger `handleSave`.
14. Suggestion logic stays separate from `lib/ai.ts`.

## Remaining Risks

| Risk | Severity | Notes |
|---|---|---|
| Suggestion quality / hallucinated domains | **Medium** | The model may suggest plausible-but-wrong or non-existent domains. Mitigations: every suggestion passes `validateCompetitorUrl` (format/SSRF only — does not check the site exists or is truly a competitor); suggestions are clearly labelled "AI suggestions, not verified competitors"; the user must confirm each one; and a bad URL simply fails at scrape time and is shown as "Failed / not used" (Stage 2). Reachability is intentionally not verified to keep the flow fast and cheap. |
| AI cost / abuse of the suggest endpoint | Low (mitigated) | A per-user cost guard (`checkRateLimit`, 10/hour) now runs **before** the AI call and returns a 429 on exceed. **Caveat:** the limiter is in-memory and per-process, so it is not shared across serverless instances — a determined user hitting multiple instances could exceed 10/hour overall. It is a soft cost guard against loops/accidental spam, not a hard quota. If a hard cross-instance cap becomes necessary, move to a DB- or KV-backed counter (would add storage). |
| No live UI visual verification | Low | App not started against live AI/DB this stage. Logic is covered by behavioral + structural tests and typecheck/build. The suggest panel and merge logic were not click-tested in a browser. |
| Suggestion validation is format-only | Low | `validateCompetitorUrl` blocks private/internal/credentialed URLs and bad formats, but does not confirm the domain resolves or is genuinely a competitor. Acceptable: same trust level as manual entry, and Stage 2 surfaces scrape failures transparently. |
| 2 pre-existing smoke-test failures | Low | Unrelated string-assertion drift. Fix in test-maintenance pass. |
