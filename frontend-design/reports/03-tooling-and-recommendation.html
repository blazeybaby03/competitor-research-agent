FRONTEND DESIGN STRATEGY - PART 3 OF 3

Tooling Recommendations & Final Launch Plan

Practical guidance for implementing the selected frontend direction with Codex or Claude Code. The priority is public launch speed, revenue safety, and production stability, not a pre-launch design-system migration.

Stack context

CompeteIQ already runs on Next.js 15.5 App Router, React 19, TypeScript, Tailwind CSS 3.4, lucide-react, Supabase, Stripe, ScraperAPI, and Anthropic Claude. The current app already has useful global Tailwind component classes: .btn-primary, .btn-secondary, .input, .card, .badge-trial, and .badge-active.

The best launch move is to use that existing stack first. New UI tooling can wait until the product is public and the first conversion data shows where polish matters most.

Deliberately not recommended before launch

Do not add a second component framework, another icon library, global state management, CSS-in-JS, heavy animation, or 3D/WebGL hero effects. None of those moves helps the Friday launch path, and each adds maintenance or verification surface. The current product needs clearer proof, report hierarchy, and trust copy more than a new frontend stack.

Final recommendation

Which design should I choose if I want the fastest public launch?

Draft 1 - Clear Signal. It uses the current Tailwind tokens, global component classes, lucide icons, current navigation, and existing route structure. It improves the public experience without touching backend logic, Stripe, Supabase, URL validation, or report generation.

Which design should I choose if I want the highest perceived value?

Draft 2 - Briefing Room. It makes the report feel like a paid briefing rather than a generic AI output. The "anatomy of a report", executive-summary-first layout, opportunity cards, source trust, and pricing value framing best support the $79/mo subscription.

Which design should I choose if I want the strongest long-term SaaS brand?

Draft 2 - Briefing Room. It is credible, scalable, and not dependent on gimmicks. Keep Draft 3 - Radar as a post-launch differentiator once there is enough product usage to justify gap scores, watchlists, and more distinctive interaction patterns.

Which tools should be used immediately?

  • Existing Tailwind CSS, current brand tokens, current global component classes, lucide-react, existing next/font Inter setup, existing loading/error UI patterns, and the current verification commands.
  • Use Playwright only if production UI changes need a browser smoke check or visual screenshots.

Which tools should be left for post-launch?

  • shadcn/ui, broader Radix primitives, react-hook-form + Zod migration, Recharts/Tremor, Framer Motion, Storybook, Chromatic, v0/Figma-to-code, and Lighthouse CI pipeline gates.

The recommended play

Ship Draft 1 now, aim the product toward Draft 2. The immediate implementation should refine the landing page, report hierarchy, loading states, empty states, and billing trust copy using the existing stack. After launch, upgrade the report surface into the more premium Briefing Room direction. Test Radar only after the core funnel has paying users.

Suggested implementation plan for Codex / Claude Code

Phase 0 - Scope lock

  1. Confirm the launch-safe boundaryOnly touch frontend presentation files and shared UI components needed for Draft 1. Do not change Stripe checkout/webhook logic, Supabase auth/RLS behavior, trial-credit logic, URL validation, scraping, or AI report generation.
  2. Inventory current UI filesRead app/page.tsx, dashboard pages, components/BusinessForm.tsx, components/GenerateReportButton.tsx, components/ReportView.tsx, components/DashboardNav.tsx, app/globals.css, and tailwind.config.ts.
  3. Preserve existing behaviorMake styling/content/hierarchy changes only. Any form or API behavior change must be justified as a launch blocker before implementation.

Phase 1 - Draft 1 launch UI

  1. Landing pageRefine app/page.tsx around one CTA: "Generate your first report free." Add a real-looking report preview above the fold, trust strip, clearer 3-step flow, and practical proof copy.
  2. Report viewImprove components/ReportView.tsx so completed reports open with executive summary, market gaps, and recommended actions before deeper analysis. Keep copy-report behavior intact.
  3. Generation stateImprove components/GenerateReportButton.tsx and generating report state copy with a clear step-based wait message: scraping, reading, analysing, writing. Do not fake live backend progress.
  4. Dashboard statesPolish empty and ready states in dashboard/reports pages: no reports, no competitors, trial used, Pro active, failed/generating/completed statuses.
  5. Billing trustPolish billing copy around one Pro plan, secure Stripe billing, unlimited reports, and support contact. Keep server-side price handling untouched.

Phase 2 - Verification

  1. Run automated checksRun npm run typecheck, npm run lint, npm run test, npm run build, or simply npm run verify if time allows.
  2. Manual visual checkOpen the app locally and inspect landing, signup/login entry points, dashboard, reports list, report detail, billing, and mobile widths.
  3. Report risks clearlyIf credentials or local services block runtime testing, document the exact blocker and list the manual Supabase/Stripe/Vercel checks required.

Phase 3 - Post-launch upgrade path

  1. Move toward Briefing RoomAfter launch, improve the report page with source chips, section navigation, opportunity cards, and a stronger sample-report landing showcase.
  2. Add tooling only when justifiedIntroduce shadcn/Radix for specific components, not as a full rewrite. Add Zod/forms, charts, or motion only when a customer-facing workflow needs them.
  3. Experiment with RadarPrototype gap score, watchlist, and command input as a measured post-launch feature, not part of the first public release.
